本文档基于当前代码与部署脚本（`deploy/deploy.sh`）整理，适配标准目录：`/opt/capics`。

1. 目标目录规范
服务器统一使用以下路径：
- 代码根目录：`/opt/capics`
- 后端：`/opt/capics/backend`
- 前端：`/opt/capics/capics-frontend`
- 部署脚本：`/opt/capics/deploy`

## 2. 环境要求
- OS：Ubuntu 22.04+/24.04（其他 Linux 发行版可适配）
- Java：17+
- Maven：3.8+
- Node.js：18+
- PostgreSQL：14+
- Nginx：1.18+
- 账号：具备 `sudo` 权限

## 3. 拉取代码
```bash
cd /opt
sudo mkdir -p /opt/capics
sudo chown -R $USER:$USER /opt/capics

# 首次
cd /opt/capics
git clone https://github.com/Sun-Frank/Capacity.git .

# 日常更新
git fetch --all
git pull --ff-only origin main
```

## 4. 配置生产环境变量
```bash
cd /opt/capics
cp deploy/.env.prod.example deploy/.env.prod
vi deploy/.env.prod
```

至少确认以下参数：
- 数据库：`DB_HOST/DB_PORT/DB_NAME/DB_USERNAME/DB_PASSWORD`
- 管理员账号：`POSTGRES_ADMIN_USER/POSTGRES_ADMIN_PASSWORD`
- 安全：`JWT_SECRET`（32位以上）
- CORS：`APP_CORS_ALLOWED_ORIGINS`
- Nginx：`NGINX_SERVER_NAME/NGINX_WEB_ROOT/NGINX_CONF_PATH`
- 端口：`SERVER_PORT`（默认 8080）
- 模式：`DB_SCHEMA_MODE`

### DB_SCHEMA_MODE 说明
- `incremental`（推荐生产）：只执行 `deploy/sql/migrations/*.sql` 中未执行过的脚本
- `bootstrap`：仅在空库时执行 `schema.sql`
- `reset`：强制执行 `schema.sql`（会重建表，不建议生产）

## 5. 一键部署
```bash
cd /opt/capics
bash deploy/deploy.sh deploy/.env.prod
```

脚本会完成：
1. 后端打包（Maven）
2. 前端构建（Vite）
3. 初始化数据库角色/库
4. 按 `DB_SCHEMA_MODE` 执行 schema/migration
5. 发布后端 jar 到 `/opt/capics/backend/capics-backend.jar`
6. 写入环境文件 `/opt/capics/backend/capics-backend.env`
7. 配置并重启 `capics-backend`（systemd）
8. 发布前端到 `NGINX_WEB_ROOT`
9. 写入 Nginx 站点并重载

## 6. HTTPS 配置
`deploy.sh` 默认 `ENABLE_HTTPS=true`，会写入 80->443 跳转与 443 站点。

需要提前准备证书：
- `SSL_CERT_PATH`
- `SSL_CERT_KEY_PATH`

若未准备好证书，可临时设置：
```dotenv
ENABLE_HTTPS=false
```

## 10. 常见问题
### 10.1 登录后无响应 / CORS 错误
- 检查 `APP_CORS_ALLOWED_ORIGINS` 是否包含实际访问域名（含 http/https）
- 检查 Nginx 是否将 `/api/` 正确反代到 `127.0.0.1:8080`

### 10.2 静态保存报 413
当前版本已改为“元数据保存 + 后端重算”。若仍出现 413：
1. 确认前端是最新包（包含 `saveMode: 'server-recompute'`）
2. 确认后端是最新 jar
3. 检查 Nginx `client_max_body_size`（建议 256m）

### 10.3 模板下载 403
- 确认认证 token 有效
- `/api/lines/template`、`/api/ct-lines/template` 已在后端放行，其他模板仍需登录

## 11. 回滚建议
部署前保留旧 jar 与旧前端静态目录；异常时可快速回滚：
- 回滚后端 jar 并重启 `capics-backend`
- 回滚前端静态目录并 reload Nginx
