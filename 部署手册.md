## 1. 系统概述

CAPICS 是一个生产产能评估系统，包含：

- **后端**：Spring Boot 2.7 + JPA + PostgreSQL（端口 8080）
- **前端**：Vue 3 + Vite（构建后由 Nginx 托管）
- **功能模块**：静态产能评估、动态产能模拟（支持快照保存/加载）、仪表盘

### 1.1 模块开发状态

#### 已完成模块

| 模块 | 路由路径 | 说明 |
|---|---|---|
| 仪表盘 | `/` | 产线 LOAD 汇总表，支持静态/动态数据源切换，快照加载 |
| 产品管理 | `/products` | 编码族、产品主数据的导入与管理 |
| MRP 计划 | `/mrp` | MRP 数据导入、周/月报查询 |

| 静态产能评估（周） | `/capacity-assessment` | 按周维度计算产线 LOAD，支持导入核算 |
| 动态产能模拟（周实时） | `/capacity-realtime` | 实时调整参数（CT/OEE/人数/需求量），支持快照保存 |

### 1.2 数据流向说明

```
                        ┌─────────────────┐
                        │   Excel 导入    │
                        └───────┬─────────┘
                                │
          ┌──────────┬──────────┼──────────┬──────────────┐
          ▼          ▼          ▼          ▼              ▼
    ┌──────────┐ ┌───────┐ ┌────────┐ ┌──────────┐ ┌──────────┐
    │ 编码族   │ │ 产品   │ │MRP计划 │ │工艺路线  │ │生产线配置│
    │(family) │ │(product│ │(mrp)   │ │(routing) │ │(line_cfg)│
    └────┬─────┘ └───┬───┘ └───┬────┘ └────┬─────┘ └────┬─────┘
         │            │         │           │             │
         └────────────┴────┬────┴───────────┴─────────────┘
                            │ 所有基础数据齐全后
                            ▼
               ┌────────────────────────┐
               │    静态产能评估         │  ──►  Dashboard（静态）
               │  (capacity-assessment) │
               └───────────┬────────────┘
                           │ 用户实时调整参数后
                           ▼
               ┌────────────────────────┐
               │   动态产能模拟          │  ──►  保存快照
               │(capacity-realtime)    │      (simulation_snapshot)
               └───────────┬────────────┘
                           │
                           ▼
               ┌────────────────────────┐
               │      仪表盘             │
               │     Dashboard          │  ──►  加载快照（动态）
               └────────────────────────┘
```

**数据依赖顺序（必须按此顺序导入）**：

```
编码族（family） → 产品（product） → 编码族定线（family_line）
                                    → 工艺路线（routing + routing_item）
      ↑                                        │
      │                                        ▼
      └─────── 生产线配置（line_config） ◄─────┘
                          │
                          ▼
                   MRP 计划（mrp_plan）
                          │
                          ▼
              ┌────────────┴────────────┐
              │  静态产能评估 / 动态产能模拟  │
              └────────────┬────────────┘
                           │
                           ▼
                      仪表盘展示
```

**关键说明**：

- **编码族**是最上游的基础数据，产品、工艺路线均依赖它
- **生产线配置**（工作天数/班次/工时）是产能计算的 denominator，任何时候都可单独调整
- **MRP 计划**提供需求量（demand）数据，是 LOAD 计算的核心输入
- **工艺路线**建立"成品 → 组件 → 产线"的映射关系，决定需求被分配到哪条产线
- **动态产能模拟**不修改任何数据库数据，只修改前端内存中的计算参数，保存快照时才持久化结果
- **仪表盘**只读取和展示数据，不产生任何原始数据

### 1.3 产能计算核心逻辑（编码族定线的作用）

以下说明选定一个 MRP 版本后，系统如何将成品的需求量拆分并分配到具体产线的全过程：

#### 步骤一：根据 MRP 版本找到所有成品

```
MRP Plan 表（version = 'v1'）
  └─ 成品物料号 A ─── 需求量 qty_a（按周/按月）
  └─ 成品物料号 B ─── 需求量 qty_b
  └─ ...
```

#### 步骤二：展工艺路线（BOM 展开）

```
成品物料号 A
  └─ 工艺路线（routing）
        └─ 组件 a1 ─── BOM层级 1 ─── 关联产线 L1
        └─ 组件 a2 ─── BOM层级 2 ─── 关联产线 L2
        └─ ...
  └─（每条工艺路线都可能有多条，形成一棵 BOM 树）
```

#### 步骤三：查组件的编码族

```
产品表（product）
  └─ 组件 a1 → family_code = 'F001'
  └─ 组件 a2 → family_code = 'F002'
```

> 组件本身记录了它属于哪个编码族（family_code）。

#### 步骤四：编码族定线（关键一步）

```
编码族定线表（family_line）
  └─ family_code='F001' → 实际生产线 line_code='SMT1001N'
  └─ family_code='F002' → 实际生产线 line_code='DIP1001N'
```

**编码族定线是"编码族"到"实际产线"的桥梁**。
同一个编码族可以对应多条不同的产线，定线表确定最终用哪条产线生产。

#### 步骤五：获取产线参数，计算 LOAD

```
编码族表（product_family）
  └─ family_code='F001' + line_code='SMT1001N' → pf='PF01'

生产线配置表（line_config）
  └─ line_code='SMT1001N' → 工作天数5 / 每天2班 / 每班8h / OEE=85%

产品表（该产线下此组件）
  └─ CT=30s / OEE=85 / 班人数=8
```

**LOAD 计算公式**：
```
LOAD = (需求量 × CT) / (工作天数 × 每天班数 × 每班时长(h) × OEE/100 × 3600)

示例（周）：
LOAD = (1000 × 30) / (5 × 2 × 8 × 0.85 × 3600)
    ≈ 0.103 （即 10.3% 利用率）
```

#### 汇总输出到仪表盘

```
产线 SMT1001N：
  ├─ 组件 a1  ─── 周W01 LOAD: 0.103
  ├─ 组件 a2  ─── 周W01 LOAD: 0.215
  └─ 合计      ─── 周W01 LOAD: 0.318

产线 DIP1001N：
  └─ ...
```

> LOAD > 1.0（即 100%）表示产能超负荷，仪表盘会标红警告。

## 2. 服务器环境要求

- **操作系统**：CentOS 7 / Ubuntu 20.04 及以上
- **Java**：JDK 11（如已安装 OpenJDK 11 可用 `java -version` 确认）
- **Node.js**：v16 及以上（仅构建前端时需要）
- **PostgreSQL**：12 及以上
- **Nginx**：1.18 及以上
- **Git**：用于拉取代码（或直接打包传输）

## 3. 数据库初始化

### 3.1 创建数据库和用户

```bash
# 登录 postgres
su - postgres
psql

# 执行以下 SQL
CREATE DATABASE capics;
CREATE USER capics_user WITH PASSWORD 'your_password';   -- 请修改为强密码
GRANT ALL PRIVILEGES ON DATABASE capics TO capics_user;
```

### 3.2 初始化表结构

项目自带建表脚本位于：`backend/src/main/resources/schema.sql`

```bash
# 进入数据库执行建表脚本
psql -U capics_user -d capics -f /path/to/schema.sql
```

> **注意**：`schema.sql` 包含完整建表语句和初始数据（管理员账号、产线配置等）。
> 每次执行会 `DROP` 已有表，**仅限首次初始化或重建数据库时使用**。

### 3.3 修改 application.yml

`backend/src/main/resources/application.yml` 中的数据库连接配置需要根据实际环境修改：

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/capics
    username: capics_user        # ← 修改为上面创建的用户名
    password: your_password      # ← 修改为上面设置的密码
    driver-class-name: org.postgresql.Driver
```

> **安全提醒**：建议后续将此密码和 JWT 密钥改为从环境变量读取，详见第 7 节。

## 4. 后端部署

### 4.1 构建

```bash
cd backend

# 清理并打包（跳过测试）
mvn clean package -DskipTests
```

构建成功后生成：`backend/target/capics-backend-1.0.0.jar`

### 4.2 创建 Systemd 服务

```bash
sudo vim /etc/systemd/system/capics-backend.service
```

写入以下内容：

```ini
[Unit]
Description=CAPICS Backend Service
After=network.target postgresql.service

[Service]
Type=simple
User=your_server_user          # 替换为实际运行用户
WorkingDirectory=/path/to/backend
ExecStart=/usr/bin/java -jar /path/to/backend/target/capics-backend-1.0.0.jar
Restart=on-failure
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

### 4.3 启动服务

```bash
sudo systemctl daemon-reload
sudo systemctl enable capics-backend
sudo systemctl start capics-backend
sudo systemctl status capics-backend
```

## 6. Nginx 配置

```bash
sudo vim /etc/nginx/conf.d/capics.conf
```

写入以下配置（假设后端运行在 8080 端口）：

```nginx
server {
    listen 80;
    server_name your_server_ip;   # 或域名

    # 前端静态文件
    root /path/to/capics-frontend/dist;
    index index.html;

    # 前端路由（Vue Router history 模式）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API 反向代理到后端
    location /api/ {
        proxy_pass http://127.0.0.1:8080/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
        expires 1y;
        add_header Cache-Control "public, no-transform";
    }
}
```

验证并重载配置：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

## 7. 环境变量配置（重要）

> **安全要求**：数据库密码和 JWT 密钥**禁止硬编码**在配置文件中交付给第三方。
>
> 以下为可选但**强烈推荐**的改进方式，由继承者执行：

### 7.1 将敏感信息改为环境变量

修改 `application.yml`：

```yaml
spring:
  datasource:
    password: ${DB_PASSWORD}    # 从环境变量读取

jwt:
  secret: ${JWT_SECRET}
```

### 7.2 在服务器上设置环境变量

```bash
# 写入系统环境变量（永久）
echo 'export DB_PASSWORD=your_secure_password' >> /etc/profile
echo 'export JWT_SECRET=your-256-bit-secret-key-must-be-long-enough' >> /etc/profile

# 使生效
source /etc/profile

# 验证
echo $DB_PASSWORD
```

### 7.3 通过 Systemd 服务传递环境变量

编辑 `/etc/systemd/system/capics-backend.service`：

```ini
[Service]
Environment="DB_PASSWORD=your_secure_password"
Environment="JWT_SECRET=your-256-bit-secret-key-must-be-long-enough"
ExecStart=/usr/bin/java -jar /path/to/backend/target/capics-backend-1.0.0.jar
```

然后重载：

```bash
sudo systemctl daemon-reload
sudo systemctl restart capics-backend
```

## 10. 常见问题

### Q1: 后端启动失败，提示数据库连接错误

1. 确认 PostgreSQL 已启动：`sudo systemctl status postgresql`
2. 确认数据库、用户名、密码与 `application.yml` 中一致
3. 确认 pg_hba.conf 允许本机免密登录或配置了 md5 认证

### Q2: 前端页面空白或资源 404

1. 确认 `dist/` 目录已正确部署到 Nginx `root` 指定的路径
2. 确认 `nginx.conf` 中 `try_files` 配置正确（支持 Vue Router）
3. 查看 Nginx 错误日志确认具体原因

### Q3: API 请求返回 401/403

1. 确认后端已正常启动
2. 确认 JWT 密钥与前端一致（`localStorage` 中的 token）
3. 尝试清除浏览器缓存或重新登录

### Q4: 快照功能无法使用

1. 确认 `simulation_snapshot` 表已创建（执行过建表脚本）
2. 确认数据库用户有该表的读写权限

### Q5: 导入功能失败

1. 确认 Excel 文件格式为 `.xlsx`（不支持 `.xls`）
2. 确认编码族/产品等基础数据已先导入（如工艺路线依赖编码族数据）
3. 工艺路线导入时，同一成品的组件行必须紧跟在成品行之后，且成品行 Product Number 必填
4. MRP 导入的 Version 字段必填

### Q6: 导入后数据没有变化

1. 导入编码族/产品时，系统以 `family_code + line_code` 为主键判断新增或更新
2. 重复导入会覆盖已有数据（主键相同则更新）
3. 产品导入的其他字段（CT/OEE/人数等）从编码族继承，不会单独更新

## 11. 数据导入模板

系统提供 5 个预制的 Excel 导入模板，位于：`backend/import_templates/`

| 模板文件名 | 导入类型 | 必填列 | 表头方式 | 说明 |
|---|---|---|---|---|
| 编码族导入模板.xlsx | 编码族 | familyCode, lineCode | 支持中英文灵活表头 | 其他字段可选，为空则不更新 |
| 产品导入模板.xlsx | 产品主数据 | itemNumber, familyCode | 固定位置 | CT/OEE/人数等字段从编码族继承 |
| 编码族定线导入模板.xlsx | 编码族定线 | familyCode, lineCode | 固定位置 | 建立编码族与生产线的关联关系 |
| MRP导入模板.xlsx | MRP 计划 | Item Number, Version | 固定位置 | 按 A~J 十列顺序填写 |
| 工艺路线导入模板.xlsx | 工艺路线 | Product Number*, Component Number*, Line Code*, BOM Level* | 固定位置 | 同成品号的多行组件要紧邻成品行下方 |

> *工艺路线的 Product Number 在成品行必填，组件行可只填 Component Number / Line Code / BOM Level。

**导入顺序建议**：编码族 → 产品 → 编码族定线 → 工艺路线 → MRP（基础数据先导入，上游依赖后导入）。

### 各模板列说明

**MRP 导入模板（A~J 十列）**：
- A: Item Number（必填）
- B: Description
- C: Site
- D: Production Line
- E: Release Date（建议 Excel 日期格式）
- F: Due Date
- G: Quantity Scheduled
- H: Quantity Completed
- I: Routing Code
- J: Version（必填）

**工艺路线导入模板**：
- A: Product Number（必填，成品行填）
- B: Description
- C: Component Number（必填）
- D: Line Code（必填）
- E: BOM Level（必填）

## 12. 数据库备份与恢复

### 12.1 备份

```bash
# 全量备份（建议定期执行）
pg_dump -U capics_user capics > backup_$(date +%Y%m%d).sql

# 备份并压缩
pg_dump -U capics_user capics | gzip > backup_$(date +%Y%m%d).sql.gz
```

> 项目目录下的 `backend/database_backup.sql` 为当前开发环境的数据库完整备份，
> 交接时可一并提供（内含测试数据）。

### 12.2 恢复

```bash
# 恢复前需先创建空库
psql -U capics_user -d capics < backup_20250101.sql
```

### 12.3 定时自动备份（可选）

```bash
# 添加 crontab 任务，每天凌晨 3 点自动备份
crontab -e

# 写入以下内容：
0 3 * * * pg_dump -U capics_user capics > /path/to/backup/capics_$(date +\%Y\%m\%d).sql 2>> /path/to/backup/cron.log
```
